# Design: `14.1.1-pkgnec.1` — canary release to prove the OIDC trust chain

**Status:** approved
**Date:** 2026-07-11
**Author:** Hardy Nguyen (maw.signup@gmail.com)

## Context

`@pkg-nec/http-server@14.1.1-pkgnec.0` was published by hand (a manual `npm publish` from a laptop, established during the identity-handshake release). The follow-up OIDC spec then wired GitHub Actions to publish future versions automatically via short-lived OIDC tokens, gated on a GitHub Environment approval, with signed provenance attestations. That machinery is in place on `trunk`, verified statically (config exists, env exists, YAML parses), but has never fired end-to-end for real.

The next planned release is `14.1.2` — the first substantive vulnerability patch of this fork. `14.1.2` has real stakes: it fixes CVEs users are waiting on. Discovering a broken OIDC handshake, a missing `id-token` permission, or a mis-registered Trusted Publisher field DURING `14.1.2`'s publish would be a bad moment. This canary spec exists to move that discovery earlier, to a release with no stakes at all.

`14.1.1-pkgnec.1` is a canary in the classical sense: a real published version, with real provenance, whose only functional difference from `14.1.1-pkgnec.0` is its version number. If the pipeline breaks, this is where it breaks.

## Goal

Fire `.github/workflows/publish.yml` end-to-end for the first time by publishing `@pkg-nec/http-server@14.1.1-pkgnec.1` as a `canary`-tagged release, and confirm the four load-bearing behaviors of the pipeline:

1. The tag → GitHub Release → workflow trigger chain fires.
2. The `npm-publish` environment approval gate pauses the job as intended.
3. The OIDC handshake with npm succeeds (no `E401`).
4. Provenance attestation gets recorded and is viewable on the published version's npm page.

## Non-goals

- **No code changes.** Zero. The tarball's `lib/`, `bin/`, and `doc/` contents must be byte-identical to `14.1.1-pkgnec.0`, which is itself byte-identical to upstream `v14.1.1`. `git diff v14.1.1-pkgnec.0 HEAD -- lib bin public` must remain empty on the release branch.
- **No `latest` dist-tag movement.** `14.1.1-pkgnec.1` publishes with dist-tag `canary`. `latest` stays at `14.1.1-pkgnec.0`. Users installing `@pkg-nec/http-server` with no version continue to get `.0` (subject to `min-release-age`).
- **No dep changes**, no lockfile drift, no `engines.node` bump.
- **No modification to `.github/workflows/publish.yml`.** The workflow is the artifact under test — modifying it during the test invalidates the test.
- **No modification to the npm Trusted Publisher config or the GitHub Environment.** Same reason.

## Definition of done

1. Git tag `v14.1.1-pkgnec.1` exists on `trunk` and is pushed to `origin`.
2. GitHub Release for that tag exists (not draft).
3. `publish.yml` workflow ran to completion for that Release, including a triggered environment approval that was granted.
4. `npm view @pkg-nec/http-server dist-tags` (run from outside this repo to bypass `.npmrc`) shows `latest: 14.1.1-pkgnec.0` AND `canary: 14.1.1-pkgnec.1`.
5. `npm view @pkg-nec/http-server@14.1.1-pkgnec.1` returns metadata for the published version, including a non-null `dist.attestations` field.
6. `https://www.npmjs.com/package/@pkg-nec/http-server/v/14.1.1-pkgnec.1` shows a "Provenance" section listing the source repo (`pkg-nec/http-server`), the workflow (`.github/workflows/publish.yml`), and the commit SHA the tarball was built from.
7. The published version is deprecated via `npm deprecate` with a message identifying it as the smoke-test canary.

If any of items 1–6 fails, the canary has done exactly what it's here to do: surface a specific failure of the OIDC trust chain BEFORE `14.1.2` depends on it. Item 7 is the cleanup after success.

## Work items

### 1. In-tree changes (single commit, on branch `chore/canary-14.1.1-pkgnec.1`)

Two edits to `package.json`:

- `version`: `14.1.1-pkgnec.0` → `14.1.1-pkgnec.1`
- `publishConfig`: add `"tag": "canary"` alongside the existing `access` and `provenance` fields

Final `publishConfig` block:

```json
"publishConfig": {
  "access": "public",
  "provenance": true,
  "tag": "canary"
}
```

That is the entire in-tree diff.

### 2. Branch → PR → merge

Standard trunk-based flow already used for prior releases:

- `git push -u origin chore/canary-14.1.1-pkgnec.1`
- `gh pr create --base trunk`
- Wait for `Test (16.x, ubuntu-latest)` to go green on the PR
- Merge to `trunk`

CI is expected to pass identically to the OIDC PR — no code changes, no lockfile change.

### 3. Publish sequence (post-merge, from trunk)

Executed from a clean checkout of `trunk` at the merge tip:

```bash
git checkout trunk
git pull --ff-only origin trunk
git tag -a v14.1.1-pkgnec.1 -m "canary smoke test of OIDC publish workflow"
git push origin v14.1.1-pkgnec.1
gh release create v14.1.1-pkgnec.1 \
  --title "v14.1.1-pkgnec.1 — OIDC pipeline canary" \
  --notes "Smoke-test release. Byte-identical to v14.1.1-pkgnec.0 under lib/, bin/, doc/. Exists solely to fire the OIDC publish workflow end-to-end before 14.1.2's real vulnerability patch depends on it. Will be deprecated post-verification. Not for production consumption."
```

`gh release create` transitions the Release from "not created" straight to "published" (no `--draft` flag), which fires `publish.yml` on `release: published`.

Then, in the GitHub Actions UI:

- Open the running workflow at `https://github.com/pkg-nec/http-server/actions`
- The `publish` job pauses on "Waiting for review" against the `npm-publish` environment
- Click **Approve**
- Watch the job resume, run `npm ci`, run the tag-vs-version sanity check (should pass — tag `v14.1.1-pkgnec.1` matches `package.json` version), and finally run `npm publish --provenance`

The publish step's log should end with something like `+ @pkg-nec/http-server@14.1.1-pkgnec.1` and a URL to the provenance attestation on Sigstore's transparency log.

### 4. Verification

Executed from any directory OUTSIDE this repo to bypass the local `.npmrc`'s `min-release-age=7`:

```bash
cd /tmp

# DoD 4: dist-tags
npm view @pkg-nec/http-server dist-tags --json
# expected: {"latest":"14.1.1-pkgnec.0","canary":"14.1.1-pkgnec.1"}

# DoD 5: version and attestations
npm view @pkg-nec/http-server@14.1.1-pkgnec.1 --json | node -e "
  let s='';process.stdin.on('data',d=>s+=d);
  process.stdin.on('end',()=>{
    const p=JSON.parse(s);
    console.log('version:', p.version);
    console.log('has attestations:', !!p.dist?.attestations);
  });
"
# expected:
#   version: 14.1.1-pkgnec.1
#   has attestations: true
```

DoD 6 is human-visual: open the npm page and confirm the Provenance section renders.

### 5. Post-verification cleanup

```bash
npm deprecate @pkg-nec/http-server@14.1.1-pkgnec.1 \
  "smoke-test canary used to verify the OIDC publish pipeline; use @pkg-nec/http-server@latest for the current release"
```

Then confirm from `/tmp`:

```bash
npm view @pkg-nec/http-server@14.1.1-pkgnec.1 deprecated
# → smoke-test canary used to verify ...
```

## Follow-up (out of scope for this spec, called out to prevent loss)

Before `14.1.2`'s release ships, `"tag": "canary"` MUST be removed from `package.json`'s `publishConfig`. If left in place, `14.1.2` would also publish under the `canary` dist-tag and `latest` would remain stuck at `14.1.1-pkgnec.0` even after a real patch shipped. This is a one-line chore on the `14.1.2` release prep branch — flagged here so it does not get forgotten during vulnerability triage. The 14.1.2 spec should reference this note explicitly.

## Risks & mitigations

- **`latest` accidentally moves to canary.** Prevented by `publishConfig.tag: "canary"`. Verified by DoD 4 (dist-tags read from outside the repo). Detection: the DoD verification itself. Fix if it did happen: `npm dist-tag add @pkg-nec/http-server@14.1.1-pkgnec.0 latest` restores `latest`.
- **OIDC handshake fails at publish (`E401`).** Would mean the Trusted Publisher config on npm doesn't match what the workflow presents in its OIDC JWT. Debug by re-reading the four Trusted Publisher fields (organization, repository, workflow filename, environment name) in the npm UI and comparing to the workflow's context. This is precisely the failure mode the canary is here to catch — the workflow logs will name the specific claim that failed to match.
- **Provenance attestation missing after publish.** Would mean either the `id-token: write` permission got dropped from the workflow, the `--provenance` flag got dropped from the CLI, or `publishConfig.provenance: true` got clobbered. Debug via the workflow's publish-step log — it emits an attestation URL on success and a clear error on failure.
- **Environment approval sits waiting.** No time pressure. The workflow pauses at the environment gate until approved. GitHub Actions does have a maximum wait time for pending approvals, but it's far longer than any realistic delay for this smoke test. No cost impact while waiting.
- **`gh release create` accidentally fires the workflow before you're ready to watch.** The workflow just runs; it stops at the environment approval gate regardless of when you approve. No harm from watching late.
- **A stray label on the PR bumps the version via release-drafter.** Non-issue: release-drafter was deleted in the OIDC PR. No auto-drafts anymore.
- **`min-release-age=7` complicates verification for users other than us.** True but irrelevant for the smoke test — we verify via `npm view` from outside the repo, which bypasses the local `.npmrc` and reads registry state directly. Documented in the plan.
- **`npm deprecate` cannot be performed via OIDC (no automation path today).** Requires a logged-in npm session with rights on the `@pkg-nec` org. That's fine — deprecation is a one-time cleanup, run manually. Documented in the plan.

## Rollback

- **Workflow fails at OIDC handshake, no version reaches npm:** nothing published, nothing to roll back. Fix the mismatch (Trusted Publisher config or workflow), delete the failed GH Release + tag if you want a clean redo, then re-fire.
- **Version publishes but provenance is missing or broken:** deprecate immediately with a message flagging the failure (`npm deprecate ... "provenance attestation missing; investigate before further releases"`), then investigate.
- **Everything works but you change your mind about keeping the canary version:** `npm unpublish @pkg-nec/http-server@14.1.1-pkgnec.1` is available within 72 hours of publish. Prefer `deprecate` per the design's decisions; `unpublish` is only for genuinely-oops scenarios and destroys the provenance record.
