# Design: `@pkg-nec/http-server@14.1.4` — remove unused eslint, upgrade tap

**Status:** approved
**Date:** 2026-07-11
**Author:** Hardy Nguyen (maw.signup@gmail.com)

## Context

At `14.1.3`, `npm audit` reports 37 residual vulnerabilities. All of them live in the dev tree — the published tarball's runtime install is clean (`found 0 vulnerabilities`).

The dev-tree vulns break down like this:

- **4 direct dev deps** are flagged: `eslint`, `eslint-config-populist`, `request`, `tap`. Only `request` has its own CVE (Server-Side Request Forgery); the other three are "affected via transitives" — the deps they drag in have CVEs.
- **20 transitive-own-CVE deps.** Of those, **11 are bundled inside `tap@14`'s tarball** and can NOT be reached with `npm install <transitive>` or `overrides` — bundled deps are extracted from tap's own tarball and are frozen to whatever tap 14 shipped in 2021.
- **9 transitive-own-CVE deps ARE reachable** via `overrides` (under `eslint` or `request` subtrees, not bundled). Those are deliberately skipped in this release — the user has decided to remove eslint outright (making 4 of the 9 disappear as a side effect) and defer `request` (the remaining 5 are all under `request`).

A dep-usage survey confirmed that **only `eslint` and `eslint-config-populist` are unused code-wise**. Every other dep — including `request` and `tap` — has real code references. `request` has ~29 references in `test/*.test.js` (its removal would be a test rewrite, out of scope). `tap` has ~31 references, but all of them use the stable API surface (`require('tap').test`, `tap.plan(...)`) that survives across tap majors.

## Semver classification

Patch release. No downstream-visible changes.

- eslint and eslint-config-populist are `devDependencies`. Downstream users never install them.
- tap is a `devDependency`. Bumping it (or its runtime Node requirement) does not affect the published tarball's `engines.node` field, does not add new user-facing capability, does not change any documented behavior.
- No changes under `lib/`, `bin/`, `public/`. No API surface change.

## Goal

Publish `@pkg-nec/http-server@14.1.4` as a patch that:

1. Removes the unused `eslint` + `eslint-config-populist` dev deps and their `.eslintrc.json` config file.
2. Bumps `tap` from `^14.11.0` to the latest 21.x, freeing the 11 bundled-in-tap transitive vulnerabilities.
3. Bumps the test-CI Node matrix from `16.x` to `20.x` (required by tap 21's engine constraint).
4. Ships as a patch through the OIDC publish workflow, moving `latest` on npm to `14.1.4`.

## Non-goals

- **No touching `request`.** Removing `request` would require rewriting the ~29 test-code references to use `undici` or `node:fetch`. That's real test-code work; explicitly out of scope for this release. Deferred to a later patch.
- **No touching prod deps.** Every prod dep has at least one active code reference in `lib/`. Any bump would be a same-major upgrade at best, which was already exhausted in `14.1.2`/`14.1.3`, or a new-major upgrade that fails the semver gates from earlier releases.
- **No `engines.node` change** on the published package. It stays `">=12"`.
- **No changes to `.github/workflows/publish.yml`.** The publish workflow is already on Node 22 + `npm@11.10.0`; unaffected by this release.
- **No changes to the npm Trusted Publisher config or the `npm-publish` GitHub Environment.**
- **No `overrides` field added to `package.json`.** The `overrides` experiments during exploration are not part of this release (were confirmed not to help against `tap@14`'s bundled deps).
- **No README / CHANGELOG additions** unless the test suite discovers something that requires them.

## Scope

### Files changed

| File | Change |
|---|---|
| `package.json` | Remove `devDependencies.eslint`. Remove `devDependencies.eslint-config-populist`. Bump `devDependencies.tap` to `^21.x`. Bump `version` to `14.1.4`. |
| `.eslintrc.json` | **Delete.** |
| `.github/workflows/node.js.yml` | `node-version: [16.x]` → `node-version: [20.x]`. |

### Not changed

- `lib/`, `bin/`, `public/`, `doc/` — no touching.
- `test/*.test.js` — no touching (unless tap 21 turns out to require a small assertion-syntax fix, in which case the smallest possible edit lands with the tap-bump commit and is flagged in that commit's message).
- All other prod deps (`basic-auth`, `chalk`, `corser`, `he`, `html-encoding-sniffer`, `http-proxy`, `mime`, `minimist`, `opener`, `portfinder`, `secure-compare`, `union`, `url-join`).
- All other dev deps (`eol`, `express`, `request`).
- `.github/workflows/publish.yml`, `.github/workflows/stale.yml`.
- `README.md`, `SECURITY.md`, `LICENSE`.
- Documentation (`docs/superpowers/**`) other than the new spec, plan, and audit-delta files this release itself creates.

## Definition of done

1. `grep -rn "eslint" bin lib public test .github` returns nothing at branch tip. (Docs may contain historical mentions.)
2. `.eslintrc.json` file does not exist at branch tip.
3. `node -e "console.log(Object.keys(require('./package.json').devDependencies))"` prints an array containing `eol`, `express`, `request`, `tap` — no eslint entries.
4. `node -e "console.log(require('./package.json').devDependencies.tap)"` prints a `^21.x` range.
5. `node -e "console.log(require('./package.json').version)"` prints `14.1.4`.
6. `.github/workflows/node.js.yml` contains `node-version: [20.x]` and does NOT contain `16.x`.
7. Local `rm -rf node_modules && npm ci && npm test` succeeds on the branch tip's Node 22.x environment (previously broken by `tap@14`; the tap bump restores local test execution).
8. CI on the PR is green on `Test (20.x, ubuntu-latest)`.
9. `npm audit --json` at branch tip shows significantly fewer vulns than `14.1.3` baseline (37). Target: eslint subtree gone (~14 line items removed) + tap-bundled subtree replaced with tap-21's cleaner subtree (~11 line items removed) → **expect ~12 or fewer remaining, all under `request` or `request`'s deep transitives**.
10. `git diff v14.1.1-pkgnec.0 HEAD -- lib bin public` is empty (as it has been through every release since the handshake — the tarball's shipped code stays byte-identical to upstream v14.1.1).
11. PR opened, CI green, merged to trunk. Tag `v14.1.4` on trunk pushed to origin. GitHub Release published from the tag. `publish.yml` produced a provenance-attested tarball; `npm view @pkg-nec/http-server dist-tags` shows `latest: 14.1.4`.

## Risks & mitigations

- **Tap 21 test API broke something.** Tests use only `require('tap')`, `tap.test(...)`, and `tap.plan(...)`. All three are the stable subset that tap has preserved across majors. If a specific assertion syntax has changed (unlikely for these three), the fix is small and lands with the tap-bump commit. **Mitigation:** run `npm test` locally on the developer's Node 22 environment after the tap bump but before pushing. Node 22 currently can't run `tap@14` at all, so the first `npm test` on tap 21 IS the smoke test for API compatibility.
- **`.eslintrc.json` deletion breaks something we didn't find.** No `lint` script exists. No CI job invokes eslint. `git grep eslint` on the branch verifies no dangling references. **Mitigation:** post-removal, run `grep -rn "eslint" bin lib public test .github` and confirm empty.
- **Node 20 EOL (April 2026).** Acknowledged and accepted. Node 20 is the "minimum floor we still officially test against"; many enterprises continue to run it. Test-CI on Node 20 does NOT require Node 20 to still be receiving security patches — it only requires the runtime to install and run our test suite, which it does. If Node 20 support ever proves untenable, a future patch can bump to Node 22. Not this release's problem.
- **`min-release-age=7` in `.npmrc` is silently ignored on Node 20's bundled npm.** Documented in `.npmrc`'s own comment. `min-release-age` is npm 11+ only. Node 20 ships with npm 10.9. Test CI's `npm ci` will not honor the 7-day quarantine — but the purpose of `min-release-age` is to guard against just-published malicious packages during human dev-machine installs, not CI. CI installing without the quarantine is fine for our purposes.
- **Large `package-lock.json` diff.** Expected. The eslint subtree (dozens of entries) disappears entirely; tap's new bundled tree replaces tap's old bundled tree. Total lockfile diff will be substantial. Per-topic commits keep reviewer's mental model organized: `package.json` diff in the tap-bump commit reads as one changed range; the massive lockfile churn is downstream of that.
- **`.npmrc` supply-chain hardening might block the tap install on dev machines.** Developer laptops running `npm@11.10.0` honor `min-release-age=7`. If tap's latest 21.x patch was published within the last 7 days, the install would fail with `No matching version found for tap@^21.x`. **Mitigation:** if this happens, pick a slightly older tap 21.x version that's more than 7 days old. Not a real risk given tap's release cadence.

## Rollback

- **Single-commit revert.** The plan's per-topic commits (eslint removal, tap bump, CI Node bump, version bump) revert independently. If the tap bump surfaces a test-code problem that's not quickly fixable, revert commit 2 and ship a smaller `14.1.4` containing just the eslint removal + CI Node bump. That still cleans up the eslint subtree; the tap upgrade defers to `14.1.5`.
- **Post-publish.** If a regression reaches npm, `npm deprecate @pkg-nec/http-server@14.1.4 "<reason>"` and prepare `14.1.5` with the fix. Standard practice from prior releases.

## Follow-on work (out of scope, tracked here so nothing gets lost)

- The 9 override-reachable transitive vulns (`ajv`, `cross-spawn`, `form-data`, `mem`, `qs`, `tmp`, `tough-cookie`, `uuid`, `yargs-parser`) are deliberately skipped in this release. Once eslint is gone, some of them (`ajv`, `cross-spawn`, `tmp`, `mem`, `uuid`, `yargs-parser`) may disappear automatically because their parent subtrees came in via eslint. The remaining ones under `request` (`form-data`, `qs`, `tough-cookie`) stay until `request` itself is removed in a later patch.
- The `request` removal is deferred. Estimated ~30 lines of test-code migration to `undici` or `node:fetch`. Own patch release when ready.
- Group D of the pre-existing 15.0.0 discussion (`corser`, `http-proxy@1`, `mime@1`, `union`, `secure-compare`) is unaffected by this release. All are still active prod deps, still uncorrelated with current CVEs, still deferred to whenever a real forcing function shows up.
