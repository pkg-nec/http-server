# Design: `@pkg-nec/http-server@14.1.1-pkgnec.0` — identity handshake release

**Status:** approved
**Date:** 2026-07-11
**Author:** Hardy Nguyen (cnguyenduy@nrchealth.com)

## Context

This repo is a fork of [`http-party/http-server`](https://github.com/http-party/http-server), taken because the upstream project has become effectively unmaintained. The fork's mandate is narrow: **fix vulnerabilities and publish replacement versions.** No new features. The fork lives at `pkg-nec/http-server` on GitHub, is developed trunk-based on the `trunk` branch (branched from upstream tag `v14.1.1`), and publishes to npm under the scope `@pkg-nec`.

The overall release progression is:

- **`14.1.1-pkgnec.0`** — identity handshake (this spec). Ownership metadata only; source code identical to upstream `v14.1.1`.
- **`14.1.2`** — first real patch: vulnerability fixes via version bumps of existing deps, no dep swaps, no engine change. Later spec.
- **`15.0.0`** — major: bump `engines.node`, replace abandoned prod deps (`corser`, `union`, `http-proxy@1`, `mime@1`). Later spec.

This document covers only `14.1.1-pkgnec.0`.

## Goal

Publish a first, publicly-installable release of `@pkg-nec/http-server` that is functionally identical to upstream `http-server@14.1.1`, so that the npm scope, package name, publish credentials, and release mechanics are proven end-to-end before any code-level changes land. Also complete the ownership metadata sweep so future users, GitHub, and npmjs.com present a coherent picture of "this is a maintained fork by pkg-nec."

## Definition of done

1. `npm i -g @pkg-nec/http-server` succeeds from any environment.
2. Installed binary reports `http-server --version` → `14.1.1-pkgnec.0`.
3. `npm view @pkg-nec/http-server dist-tags.latest` returns `14.1.1-pkgnec.0`.
4. Package tarball contents match upstream 14.1.1 file-for-file under `lib/`, `bin/`, `doc/` (verified via `npm pack --dry-run` diffed against a checkout of upstream `v14.1.1`).
5. Git tag `v14.1.1-pkgnec.0` exists on `trunk` and is pushed to `origin`.
6. GitHub Release published from the tag with a short note explaining what this release is.

## Non-goals

- Any dependency upgrades (dev or prod).
- Any code changes under `lib/`, `bin/`, `public/`.
- CI matrix updates in `.github/workflows/node.js.yml`. It currently tests on Node 12–17; this is ugly but not this release's problem.
- Trusted-publishing OIDC workflow. Deferred to a follow-up spec before `14.1.2`.
- `engines.node` bump. Deferred to `15.0.0`.
- Any decision about `14.1.2` or `15.0.0` beyond the version-progression context above.

## Work items

### 1. `package.json` metadata (single commit)

Edit these fields:

- `name`: `@pkg-nec/http-server`
- `version`: `14.1.1-pkgnec.0`
- `repository.url`: `git+https://github.com/pkg-nec/http-server.git`
- `bugs.url`: `https://github.com/pkg-nec/http-server/issues`
- Add `homepage`: `https://github.com/pkg-nec/http-server#readme`
- Add `publishConfig`: `{ "access": "public" }`. Required for scoped-public publishing without passing `--access public` on every publish. Kept alongside the CLI flag as belt-and-braces.

Leave untouched:

- `contributors` — original attribution stays. New maintainer entries land in the release that carries their changes, not this one.
- `engines.node: ">=12"` — engine tightening is a major-version change; belongs in `15.0.0`.
- `dependencies`, `devDependencies`, `scripts`, `files`, `bin`, `man`, `license`, `main`, `preferGlobal`, `keywords`, `description` — none affect identity handshake.

### 2. Docs sweep (separate commits, one per file)

**`README.md`** — prepend a short "Fork" section (3–5 lines) explaining this is a maintained fork of `http-party/http-server` and pointing at `https://github.com/pkg-nec/http-server/issues` for bug reports. Update the install snippet from `npm install --global http-server` to `npm install --global @pkg-nec/http-server`. All other content preserved so existing users find familiar usage docs.

**`SECURITY.md`** — full rewrite. Replace the upstream maintainer email with pkg-nec's disclosure channel (recommend enabling GitHub private security advisories on `pkg-nec/http-server` and pointing there). Replace the supported-versions table with a fresh one that only lists `@pkg-nec/http-server` versions we intend to support.

**`LICENSE`** — no change. Upstream MIT copyright stays as-is. A second copyright line for pkg-nec is optional and can be added when the first substantive code change ships; not needed for the handshake.

### 3. GitHub-side housekeeping (single commit)

- `.github/workflows/release-drafter.yml`: remove the line `if: github.repository == 'http-party/http-server'`. Currently a no-op guard on this fork; removing it lets release-drafter run on merges to `trunk`.
- `.github/FUNDING.yml`: delete. Points at the upstream maintainer.
- `.github/ISSUE_TEMPLATE/*` and `.github/PULL_REQUEST_TEMPLATE.md`: grep for `http-party` and rewrite any URLs to `pkg-nec`. Leave template structure alone.
- `.github/workflows/stale.yml`: no change. Its behavior is fine on any repo.

### 4. GitHub repo settings (outside git; UI or `gh`)

- Set `trunk` as the default branch on GitHub.
- Delete stale remote branches on `origin`: all `dependabot/*` and inherited upstream feature branches (roughly 15 branches listed in `git branch -r`). Preserve `master` for now as a read-only reference to the upstream state at fork time.
- Enable a lightweight branch protection rule on `trunk`: require PR + green CI. Skip the mandatory-review requirement for now (solo maintainer).

### 5. Publish

Executed from a clean checkout of `trunk` at the tagged commit, on a machine logged into the npm account that has publish rights on the `@pkg-nec` org:

```bash
rm -rf node_modules
npm ci                                     # honor the lockfile; must not rewrite it
npm test                                   # sanity check; should be green
npm pack --dry-run                         # inspect tarball contents before publishing
git tag -a v14.1.1-pkgnec.0 -m "identity handshake release"
npm publish --access public --tag latest   # --tag latest is critical (see risks)
git push origin trunk v14.1.1-pkgnec.0
```

Then create a GitHub Release from the tag with a note explaining what this release is (identity handshake, no code changes vs. upstream 14.1.1).

### 6. Verification (post-publish)

- Run the six items in Definition of done.
- Also: `npm audit` against the installed package's runtime dependency tree. The result should match upstream 14.1.1's audit state; this becomes the baseline for the `14.1.2` diff.

## Risks & mitigations

- **`.npmrc` leaks into the published tarball.** Current `files` field is `["lib", "bin", "doc"]`, so root-level `.npmrc` is excluded. No action needed; call it out during `npm pack --dry-run` review.
- **`package-lock.json` drift at publish time.** The recent `deps: baseline package-lock.json` commit means the lockfile is fresh. `npm ci` (not `npm install`) is used in the publish sequence, which honors the lockfile and fails on drift instead of silently rewriting it.
- **`--tag latest` on a prerelease is unusual.** Intentional. By semver, prereleases (`14.1.1-pkgnec.0`) are strictly less than the stable version they precede (`14.1.1`), so `npm publish` without `--tag` would create a `dist-tag` but not touch `latest`, and `npm i @pkg-nec/http-server` (no version) would return "no matching version." Forcing `latest` on publish makes the package installable by name alone. When `14.1.2` publishes without a `--tag` flag, npm will overwrite `latest` automatically (stable > prerelease).
- **`@pkg-nec/http-server` name already taken on npm.** Prerequisite check: `npm view @pkg-nec/http-server` should return 404 before this release ships. If it returns metadata, the org already has a package by that name and this plan needs revisiting.
- **Existing `node_modules/` in the working tree.** Not a risk if the publish sequence starts with `rm -rf node_modules && npm ci` on a clean checkout. Called out to prevent shortcuts.

## Rollback

If the publish is broken (bad tarball contents, wrong version tag, wrong dist-tag), remediate forward:

- `npm dist-tag rm @pkg-nec/http-server latest` to unpublish the dist-tag pointer.
- `npm unpublish @pkg-nec/http-server@14.1.1-pkgnec.0` is available for 72 hours after publish. After that, the version is permanent; ship `14.1.1-pkgnec.1` with the fix.
- Delete the git tag locally and on `origin` if the tarball needs to be rebuilt from a different commit.
