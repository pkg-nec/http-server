# Design: `@pkg-nec/http-server@14.1.2` — vulnerability patch release

**Status:** approved
**Date:** 2026-07-11
**Author:** Hardy Nguyen (maw.signup@gmail.com)

## Context

`14.1.1-pkgnec.0` was the identity-handshake republish of upstream `v14.1.1`, byte-identical to upstream in the shipped tarball. `14.1.1-pkgnec.1` was the OIDC smoke-test canary — deprecated, published only to prove the release pipeline. `14.1.2` is this fork's **first release with actual code (dep tree) changes** relative to upstream. Its whole purpose is vulnerability reduction: fix as much of the audit surface as possible without breaking anything for downstream users, and without introducing new logic in the code we ship.

The prior brainstorming session established two governing rules for this fork:

- **This fork ships vulnerability patches only** — no new features, no API changes.
- **Semver on our own package must be honest** — anything that would force a breaking change gets deferred to `15.0.0`.

This spec applies those rules to the dependency graph and produces the shortest defensible path to a shipped `14.1.2`.

## Semver classification (why this is a patch)

The distinction that governs release scope:

| Bump type | Downstream-visible criteria |
|---|---|
| **Patch (14.1.2)** | Bug/security fixes only. No new user-facing capability. Same public API and documented behavior. Dev-only changes always land here. |
| **Minor (14.2.0)** | New backward-compatible functionality visible to downstream users — new CLI flags, new `createServer` options, new documented capability, loosening (widening) `engines.node`, deprecation warnings. |
| **Major (15.0.0)** | Anything that would break a downstream user — tightening `engines.node`, CommonJS→ESM shift, observable behavior change, dep replacements (`corser` → `cors` etc.), API surface removals, `main`/`exports`/`bin` shape changes. |

Every change planned in this spec is either (a) a same-major dep bump inside the range already declared in `package.json` (patch by definition), or (b) a dev-only change that doesn't ship in the tarball (patch by convention). Nothing here belongs in `14.2.0` or later.

If during execution a candidate change would move to minor or major territory, execution stops on that dep and it's added to the deferred list below — no in-flight reclassification of `14.1.2`.

## Goal

Publish `@pkg-nec/http-server@14.1.2` with:

1. The maximum vulnerability count reduction achievable within the constraints of "no new logic in `lib/bin/public`, keep `tap@14`, keep Node 16.x CI, no breaking changes to downstream."
2. First non-canary run of the OIDC publish pipeline end-to-end — `14.1.2` publishes under dist-tag `latest`, moving `latest` off `14.1.1-pkgnec.0` for the first time.

## Non-goals

- **No code changes** under `lib/`, `bin/`, `public/`. `git diff v14.1.1-pkgnec.0 HEAD -- lib bin public` must remain empty on this branch through the last commit.
- **No `engines.node` change.** Stays `">=12"`. Tightening this is major-bump territory.
- **No dep replacements.** `corser` → `cors`, `http-proxy@1` → alt, `union` → alt, `mime@1` → `mime@4`, `secure-compare` → `crypto.timingSafeEqual`, `chalk@4` → `chalk@5` (ESM), `url-join@4` → `url-join@5` (ESM), `html-encoding-sniffer@3` → `html-encoding-sniffer@4` (Node 18+) — all deferred to `15.0.0`.
- **No `tap` upgrade.** `tap@14` (dev) stays. Its transitive vuln count stays unfixed for this release. Bundled with a CI Node bump into `15.0.0` prep.
- **No CI Node bump.** `node.js.yml` stays on Node 16.x.
- **No new CLI flags, no new library options, no new documented capability.** Any such addition would move the release to `14.2.0`.
- **No changes to `.github/workflows/publish.yml`, the npm Trusted Publisher config, or the `npm-publish` GitHub Environment.** Those were verified during the canary and stay.
- **No `npm audit fix --force`.** Every bump is manual, per-dep, per-commit, deliberately reviewed.

## Process — one commit per dep bump

For each dep in the order listed above (prod deps first, then dev deps), execute the gate:

1. **Check current installed version.** `npm view <dep>` for latest published; `npm list <dep>` for currently installed.
2. **Determine target within same major.**
   - Run `npm view <dep>@^<current-major> version` to see the newest same-major version.
   - If that equals the currently installed version, skip this dep — no bump available.
3. **Check whether a newer major would ALSO be safe** (only if the same-major bump is exhausted or non-existent):
   - `npm view <dep>@latest engines type` — check whether the new major:
     - has `engines.node` still satisfying `>=12` — our own declared floor. Downstream users on Node 12–15 currently CAN install this package; a transitive dep that tightens to `>=16` would surface `EBADENGINE` warnings and break those installs at runtime. CI testing on Node 16 does NOT excuse this; declared engines is a semver contract with downstream, CI matrix is a maintainer's internal check.
     - is NOT ESM-only (`type: "module"` without a CommonJS entry)
     - has no API-surface change we depend on (spot-check by reading its changelog and grep'ing our code for the imports)
   - If ALL three pass, the new major is acceptable within a patch release. Otherwise, stop.
4. **Apply the bump.**
   - Edit `package.json` to widen the range to include the new version (e.g., `^4.1.2` → `^4.1.14`).
   - `rm -rf node_modules && npm install` to update `package-lock.json`.
   - Do NOT edit `package-lock.json` by hand.
5. **Per-commit verification.**
   - `npm ci` succeeds (lockfile stays coherent)
   - `git diff v14.1.1-pkgnec.0 HEAD -- lib bin public` remains empty
   - `npm pack --dry-run` file list unchanged (20 files, same shipped set)
   - Review the `package-lock.json` diff for unexpected transitive drift — if a transitive dep jumped a major, note it in the commit message
6. **Commit.** Message format: `deps: bump <dep> from <old-installed> to <new-installed> (<GHSA-refs or "maintenance">)`. One commit per dep for reviewability.
7. **If step 3 or 5 fails on any dep:** stop on that dep, add it to the deferred list below (in-place, in this spec doc), continue to the next dep. Do not batch-bump.

Once all bumps are landed, one final commit bumps `package.json` `version` to `14.1.2`.

## Order of operations

1. Branch: `chore/release-14.1.2` off `trunk` (already created for this spec).
2. Per-dep commits in the table order — prod first, dev second.
3. Audit delta snapshot: `docs/superpowers/audits/2026-07-11-14.1.2-audit-delta.md` records the `npm audit` output before and after, plus a per-dep summary of what was bumped and what wasn't.
4. Final commit: `chore: bump version to 14.1.2`.
5. PR to `trunk`. CI must go green. User merges.
6. Post-merge: tag `v14.1.2` on `trunk`, push tag, `gh release create v14.1.2 --title "..." --notes "..."` — fires the OIDC publish workflow. User approves the `npm-publish` environment gate.
7. Verification: `npm view @pkg-nec/http-server dist-tags` shows `latest: 14.1.2` (moved off `14.1.1-pkgnec.0` for the first time). Provenance page for the new version renders.

## Definition of done

1. All 7 prod deps identified as "same-major bumpable" (`basic-auth`, `chalk`, `he`, `html-encoding-sniffer`, `minimist`, `opener`, `portfinder`, `url-join`) have per-commit bumps applied on the branch, OR have documented reasons in the audit delta for not bumping (e.g., already at range tip).
2. All 3 dev deps identified as "same-major bumpable" (`eol`, `express`, and any others still in-scope) have per-commit bumps applied on the branch, OR documented reasons.
3. The 4 explicit "defer to 15.0.0" prod deps (`corser`, `http-proxy`, `mime`, `union`) plus the 2 replacement-track ESM prod deps (`chalk` if 5, `url-join` if 5, `html-encoding-sniffer` if 4) plus `secure-compare` are documented in the audit delta as **not bumped** with reason.
4. `git diff v14.1.1-pkgnec.0 HEAD -- lib bin public` is empty at branch tip.
5. `npm audit` at branch tip shows fewer total vulnerabilities than the baseline audit snapshot from `14.1.1-pkgnec.0`. The exact delta is recorded — no target number is set, but the audit delta doc must include the before/after numbers.
6. `package.json` `version` at branch tip is `14.1.2`.
7. PR is opened, CI green, merged. Tag `v14.1.2` on trunk pushed to origin. GitHub Release published from the tag. `publish.yml` ran end-to-end and produced a provenance-attested tarball on npm.
8. `npm view @pkg-nec/http-server dist-tags` from a fresh directory (bypassing local `.npmrc`) shows `latest: 14.1.2`.

## Risks & mitigations

- **A same-major bump changes runtime behavior in a way that breaks downstream tests.** Same-major bumps are semver-safe by contract, but occasionally maintainers misbehave. Mitigation: our published tarball's `lib/`, `bin/`, `doc/` contents never change, so downstream behavior only shifts if a runtime dep changes behavior. Per-commit review of the changelog + spot-check of the `npm audit` diff will catch obvious cases. Nothing is a hard mitigation against a hostile maintainer; that risk is minimized by the `min-release-age=7` in `.npmrc` (7-day quarantine on any freshly published version, catches malicious releases before they're consumed) but not eliminated.
- **`npm install` rewrites lockfile in surprising ways.** After each bump, review the `package-lock.json` diff; if a transitive dep jumped a major unexpectedly, either accept it in the commit (with a note) or revert and try with `npm install --package-lock-only <dep>@<version>` for a leaner diff.
- **CI test suite doesn't reproduce a real regression.** Node 16 CI runs a stale test suite via `tap@14` — coverage isn't exhaustive. Mitigation: manual smoke test after merge before publishing (spawn the server with `node bin/http-server`, hit it with `curl` on a known path, confirm 200 + expected content). Documented in the plan.
- **A vuln fix requires a same-major major-version bump that WOULD be safe by our rules but wasn't caught here.** Mitigation: the process explicitly allows accepting new majors if all three gates pass. Nothing is blanket-blocked at the same-major boundary; we just default to the safer path.
- **`publishConfig.tag: "canary"` accidentally reappears in package.json.** Prevented by the removal PR (#6, already merged). Belt-and-braces: verify at branch tip with `node -e "console.log('tag' in require('./package.json').publishConfig)"` — must print `false`. Add to per-commit verification if it ever regresses.
- **The OIDC pipeline fails again.** The canary caught two failures (Node 16's npm 8, then Node 22's npm 10.9); both were fixed. If a third failure surfaces on `14.1.2`'s publish, we treat it as the canary would have — investigate, fix, retry. The version can't reach npm until the pipeline succeeds, so no half-shipped state.

## Rollback

- **A per-dep commit turns out to be wrong** (behavior change caught in review): revert just that commit, continue with the rest. No release-wide unwind.
- **The whole release turns out to be wrong** after publish: `npm deprecate @pkg-nec/http-server@14.1.2 "regression; revert to 14.1.1-pkgnec.0"`, and prepare `14.1.3` with the fix. `npm unpublish` is not used (72-hour window and it destroys provenance history).
- **The audit delta shows a NEW vuln introduced by a bump**: revert that specific commit, skip the dep, re-run audit, continue.

Recorded here so nothing is lost during vulnerability triage. The `15.0.0` spec will consume this list.

- `corser` → `cors` (or removal if we don't need CORS at all)
- `http-proxy@1` → `http-proxy-3` or removal
- `union@0.5` → connect/express-style middleware chain, or removal
